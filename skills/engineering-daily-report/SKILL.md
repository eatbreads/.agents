# Engineering Daily Report

## Overview

根据当前对话、代码操作、已知上下文和用户补充信息，整理一份真实工程师风格的开发日报。

默认把这份 skill 视为“多 agent 共写同一份当日日报”的工作流，而不是单次生成一份新的本地文本：

- 日报标题默认只使用当天日期
- 每次写日报前，先检查飞书上是否已经存在当天对应的日报
- 如果已存在，先拉取原文，再把当前 agent 的新增内容融合进去
- 如果不存在，再创建当天的新日报
- 最终把融合后的全文覆盖回飞书原文档，而不是新增一篇重复日报

如果用户明确要“发给 mentor/发群/一小段话/极简版/简短汇报”，优先输出极简日报；但只要用户同时要求同步飞书，仍然要先执行“拉取旧日报并合并”的流程。

## Required Coordination

- 需要读写飞书文档时，必须参考 `$lark-doc` 的工作流，优先使用 `lark-cli docs +search`、`lark-cli docs +fetch`、`lark-cli docs +update`
- 如果涉及认证、权限、身份切换，先参考 `$lark-shared`
- 不要自己发明飞书文档更新方式；优先沿用飞书 skill 已定义的 shortcut 和约束

## Workflow

### 1. 先确定日报归档键

先确定“今天是哪一天”，并把这一天作为日报的唯一归档键。

默认规则：

- 标题只写日期，例如 `2026-04-03`
- 不再把任务名、项目名、agent 名写进标题
- 多个 agent 在同一天内都写到同一份日报里

如果用户明确指定了别的日期，就用用户指定的日期；否则使用当前本地日期。

### 2. 先去飞书查有没有当天日报

在生成任何新内容前，先查询飞书中是否已存在以该日期命名的日报。

优先顺序：

- 如果用户给了固定文档链接或 token，直接对该文档执行抓取
- 如果用户给了日报目录、知识库、文件夹或日报集合，但没给具体文档，先用 `$lark-doc` 的搜索能力按日期查找
- 如果用户没有给出飞书位置，只能先整理本地草稿，并明确说明尚未定位日报文档，暂时不能回写飞书

当搜索到多个同日期候选文档时，优先选择：

- 用户明确指定的那个
- 已知日报目录下的那个
- 最近更新时间最新、且标题最精确匹配日期的那个

不要在没检查飞书是否已有日报的前提下直接新建文档。

### 3. 拉取已有日报正文

如果当天日报已存在，先把正文完整拉下来，再开始融合当前 agent 的内容。

这里的目标不是“参考一下”，而是把旧文当作这次更新的 base version。

必须保留旧日报里其他 agent 已写入的有效内容，包括：

- 已有的工作项
- 已记录的问题、进展、结论
- 已存在的后续计划

不要因为当前 agent 的内容更完整，就把别的 agent 已写的信息整体抹掉。

### 4. 只整理当前 agent 本轮新增信息

只从当前会话、当前 agent 做过的操作、用户这轮明确补充的信息里提取新增内容。

优先保留：

- 当前 agent 今天实际推进了什么
- 做了哪些阅读、排查、修改、验证、设计或 review
- 搞明白了哪些关键点
- 目前还卡在哪
- 接下来这个方向准备怎么继续

不要把旧日报原文重新改写成另一篇风格完全不同的文章。先识别“当前 agent 新增了什么”，再做合并。

### 5. 把当前 agent 内容融合进旧日报

融合时遵循“增量更新，不覆盖其他 agent”的原则。

推荐合并策略：

- 优先保留旧日报中已经存在且仍然成立的信息
- 把当前 agent 的新增动作补进对应章节
- 如果旧日报缺某一节，再补新内容
- 如果当前 agent 与旧日报有冲突，以更具体、更新、更贴近实际工程状态的描述为准
- 如果发现旧日报里有明显重复表述，合并成一份更自然的表述

如果日报里需要显式体现多 agent 协作，优先在正文里自然写出“今天补充推进了哪一部分”，不要把标题改成 agent 维度，也不要把日报拆成多篇。

### 6. 再决定输出形态

如果用户没有要求极简版，默认输出完整日报。

完整日报用于：

- 飞书正式日报
- 需要可追踪工程过程的记录
- 多 agent 汇总后的日终沉淀

极简日报用于：

- 发 mentor
- 发群同步
- 临时口头汇报
- 用户明确只要一小段话

即使输出极简版，只要用户要求同步飞书，飞书里的正式日报仍优先写完整结构；极简版可以作为额外输出给用户发送。

### 7. 覆盖回飞书原文档

合并完成后，把完整日报覆盖回飞书原文档。

这里的默认动作是覆盖更新，不是追加一个新版本片段，也不是另建一篇“副本日报”。

优先使用 `$lark-doc` 中的更新能力完成覆盖写回。

如果当天日报不存在，才创建新文档；创建后也继续把它视为当天唯一日报，后续 agent 应继续拉取这篇再更新。

## Output Rules

完整日报默认使用下面结构：

```markdown
# 2026-04-03

## 1. 在做什么（背景 / 目标）

## 2. 做了哪些具体事情（重点）

## 3. 核心收获 / 理解（最重要）

## 4. 遇到的问题 / 卡点

## 5. 当前进度 & 下一步计划

## 6. 今日关键链路（可选）
```

注意：

- `#` 一级标题默认就是日期
- 不要再使用 `# 今日工作总结`
- 只有在上下文足够支撑时，才输出“## 6. 今日关键链路（可选）”

如果用户明确要求“极简版 / 发群版 / 发 mentor 版 / 一小段话总结”，改用下面结构输出给用户：

```markdown
# 今日简报

[1 段到 3 段的自然语言总结]
```

但如果用户同时要求同步飞书，飞书里的正文仍优先保留完整日报结构。

## Writing Rules

- 写得像真实工程师自己整理的日报，不要像周报模板或培训总结
- 用词自然，可以有少量口语，但不要太随意
- 优先保留工程过程，不要只写知识点
- 每一段尽量包含“做了什么 + 为什么做 + 遇到什么问题”
- 不要编造不存在的实现、修改、结论、进度
- 如果信息不足，直接说明“目前还不清楚/还没深入”
- 不要写空话，不要拔高
- 多 agent 合并后，正文要像一份自然的当日日报，不要写成互相割裂的多段 agent 日志拼盘
- 极简日报默认控制在 80 到 200 字；如果用户只要“一小段话”，优先 60 到 120 字

## Hard Bans

禁止出现这类 AI 味表达：

- 通过本次学习……
- 提升了我的能力……
- 进一步加深理解……
- 对整体有了更全面的认识……
- 为后续工作打下基础……

如果草稿里出现类似表达，删掉后重写。

## Style Heuristics

优先把日报写成“今天真实干活的记录”，而不是“知识总结”。

判断是否写对的方法：

- 让没参与的人看完后，能知道今天到底做了什么
- 能看出推进过程，而不是只有结果
- 能看出哪里已经看清，哪里还卡着
- 能看出下一步会继续往哪一层钻
- 多个 agent 连续写入后，整篇仍然像一份完整日报，而不是重复覆盖

极简日报写对的标准：

- mentor 或群里的人一眼能看懂今天干了什么
- 看得出不是泛泛看文档，而是真的在推进具体代码理解或实现
- 一段话就能看出当前进度和下一步

## Missing Context Handling

当上下文不完整时：

- 只使用当前对话中明确出现的信息
- 可以归纳，但不要脑补
- 如果信息不足以支撑某一节，就保守写法，不要凑字数
- 如果无法定位飞书日报文档，就先产出本地日报草稿，并明确说明尚未完成飞书侧拉取与覆盖
- 如果没有成功拉取到当天旧日报，不要假装已经完成合并；要明确说明当前输出是新草稿还是未验证覆盖稿
