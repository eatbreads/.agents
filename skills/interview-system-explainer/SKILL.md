# Interview System Explainer

## Overview

把抽象的技术问题讲成“能看到系统在跑”的解释，而不是只给定义。
目标是同时满足 3 件事：

- 让用户真正理解
- 让用户能复述给别人听
- 让答案带上工程意义和取舍，而不是停留在概念层

## Core Response Contract

除非用户明确要求别的格式，否则按下面结构回答：

### 1. 问题复述

先用一句人话重述用户到底在问什么，避免答偏。

### 2. 一句话本质

用一句话说清“这个东西为什么存在”。

要求：

- 不要先给教科书定义
- 优先回答“要解决什么问题”
- 这句话要能直接被用户拿去面试开场

### 3. 直觉解释

用生活类比、日常场景或具体画面把问题讲明白。

优先使用这类映射：

- pipeline 像流水线或工位加工
- TCP 像先确认再传话
- cache 像随手记在手边的小抄
- scheduler 像派单员
- queue 像待处理工单池
- retry 像失败后重新补投

不要只说“像某某”，要把映射关系说完整：

- 谁对应谁
- 数据对应什么
- 状态变化对应什么

### 4. 执行过程

这是核心部分。必须把概念讲成运行过程。

用步骤表达：

```text
1. 谁先发起
2. 谁接住请求
3. 中间怎么处理
4. 数据往哪里走
5. 状态怎么变化
6. 最后结果怎么返回
```

回答时强制覆盖下面几个问题：

- 谁在执行
- 数据怎么流动
- 状态如何变化
- 哪一步是关键分界点

避免空话，例如：

- “系统进行调度”
- “模块之间发生交互”
- “服务完成处理”

优先改成明确动作，例如：

- “scheduler 先把 job 放进队列”
- “runner 从队列取任务后拉代码并执行脚本”
- “失败状态写回存储后触发重试策略”

### 5. 工程实现

把解释落到真实系统，不要只停留在概念层。

优先补下面这类信息：

- Linux / Redis / Kafka / GitHub Actions / k8s / 数据库 / 消息队列 里通常怎么做
- 常见实现机制是什么
- 会用到哪些组件、协议、锁、队列、缓存或调度方式

如果主题是代码库或项目内问题，就结合当前上下文里的真实模块和调用链来解释，不要编成通用废话。

### 6. 关键点与 Trade-off

明确讲清：

- 为什么这么设计
- 这样设计的优点是什么
- 缺点或代价是什么
- 替代方案是什么
- 什么时候应该换另一种方案

如果是对比题，优先输出表意清晰的差异说明，例如：

- TCP vs UDP
- cache vs buffer
- 线程 vs 协程
- pipeline cache vs artifact
- 同步调用 vs 异步队列

对比时不要只列名词，要点出“核心区别发生在哪一层”。

### 7. 常见误区

如果这个问题容易被误解，补一小段“常见误区”。

优先指出：

- 用户常把哪个概念和哪个概念混在一起
- 直觉上容易想错的点是什么
- 真正影响行为的关键点是什么

### 8. 一句话总结

再压缩一次核心结论，让用户能记住。

### 9. 面试回答版

最后必须给一个可以直接说出口的精简回答：

- 先给 1 句话版本
- 再给 3 个要点

如果用户显式要求“面试版”“短答版”“30 秒版本”，优先把这部分写得更短更顺口。

## Style Rules

- 不要先讲抽象定义，先讲存在原因
- 不要堆术语，必要术语要顺手翻成人话
- 一定要有“谁在做什么”
- 一定要有流程感
- 优先让用户脑子里有画面
- 优先给可复述的表达，而不是百科式说明
- 可以专业，但不要故作高深

## Domain Adaptation

这个 skill 默认适合以下主题：

- pipeline / CI / 调度 / cache / artifact / 日志 / 重试
- 网络 / TCP / UDP / HTTP / 长连接 / 拥塞 / 重传
- 操作系统 / 进程 / 线程 / 协程 / 调度 / 内存 / IO / 锁
- 分布式 / 一致性 / 副本 / leader / quorum / 分片 / 容灾
- agent / tool calling / memory / planning / loop / retry / orchestration
- 系统设计 / 高可用 / 扩展性 / 稳定性 / 性能瓶颈

如果问题和用户当前方向相关，主动补一层工程关联，尤其是：

- AI triage
- pipeline 故障定位
- agent loop
- 自动化评审
- 稳定性与性能分析

但不要为了“关联当前方向”而偏题。主问题解释优先。

## Comparison Mode

当用户问的是“区别 / 对比 / 选型”，按下面节奏回答：

1. 先说两者本质区别
2. 再说执行路径差在哪
3. 再说工程上分别适合什么场景
4. 最后给出一句选型建议

## Good Triggers

这个 skill 适合这类请求：

- “帮我通俗解释一下 pipeline 为什么这样设计”
- “TCP 和 UDP 到底差在哪，面试怎么讲”
- “为什么 agent 要做 plan 和 retry”
- “cache 和 buffer 的区别是什么”
- “解释一下这个系统怎么跑起来的”
- “把这个八股题讲得像真的在跑代码一样”
- “给我一个好理解、面试能说出来的版本”

## Hard Bans

避免下面这些低质量回答方式：

- 一上来先贴定义
- 只说概念，不说过程
- 只说优点，不说代价
- 只说术语，不翻译成人话
- 只给教材口吻，不给面试表达
- 把不确定的实现说成确定事实

## Closing Requirement

回答结尾默认保留“面试回答版”。
如果全文已经很长，宁可压缩前面的铺垫，也要保住最后这段可复述版本。
