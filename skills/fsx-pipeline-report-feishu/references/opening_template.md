## FSX 流水线每日看护播报模板

### 目的与适用范围

本文档用于 FSX 值班场景下的每日流水线播报，而不是泛化的任意 Bits 流水线分析模板。

它适用于以下场景：

- FSX 值班期间的固定流水线巡检
- 发布前后对关键测试流水线的健康度快速确认
- 对最近一次已结束 run 的失败节点进行聚焦查看
- 将巡检结果同步到飞书电子表格并发送群消息

本模板适配当前这套看护脚本的真实产物链路：

1. 先生成三份总 CSV
2. 再写入一个新的飞书电子表格
3. 最后组织成飞书群消息

重点关注：

- 总体执行情况
- 失败用例集中在哪些 Job
- 是否存在需要值班人员继续跟进的异常点

### 统计口径

为确保看护结果的一致性，本次统计遵循以下规则：

- **统计对象**：仅统计流水线中 `jobAtom.uniqueId == "test_framework_trigger"` 且包含有效用例数字段（如 `case_num`, `failed_case`）的 Job
- **不纳入统计**：编译、部署、环境准备、清理、选择器等非测试类节点
- **最新 run 口径**：看护脚本只接受“最新且已结束”的 run；如果最新 run 仍在执行，则本轮跳过，不产出正式统计

### 数据来源与兜底

当前统计脚本按以下优先级获取用例数据：

1. 优先从 Job 的 `jobAtom.output` 中直接读取
2. 若 `output` 为空，则尝试解析 `failReason.message` 中的 JSON 内容
3. 如仍无法获取，则会进一步调用 `job_runs` 详情接口做最后补齐

### 核心指标公式

- **总通过率**：`(总成功用例数 / 总用例数) * 100%`
- **Job 失败率**：`(该 Job 失败用例数 / 该 Job 总用例数) * 100%`
- **失败贡献占比**：`(该 Job 失败用例数 / 流水线总失败用例数) * 100%`

### 建议阅读顺序

如果本次是固定五条 FSX 流水线看护结果，建议按以下顺序组织和阅读：

1. `overview_all.csv`
   先看五条流水线的总体执行情况、失败数和通过率
2. `top10_jobs_all.csv`
   再看每条流水线失败贡献最高的关键节点
3. `jobs_all.csv`
   如需继续筛选或展开，再查看全量 Job 明细

在飞书电子表格中的推荐落地顺序也应保持一致：

1. 先写 `总览`
2. 再写 `TOP10`
3. 最后写 `全量`

三段之间空 3 行，便于值班同学阅读。

### 飞书电子表格约定

当前真实交付形态是：

- 每天新建一个新的飞书电子表格
- 标题格式：`FSX 核心流水线每日看护-YYYY-MM-DD`
- 仅使用默认 `Sheet1`
- 在一个工作表中按“总览 -> TOP10 -> 全量”顺序分段写入

这样做的原因是：

- 对 CSV 天然友好
- 不依赖多维表格动态建字段
- 比 3 个独立附件更适合直接浏览

### 群消息约定

群消息建议采用以下结构：

1. 标题或一句话总览
2. 总体概览
3. 各流水线运行情况汇总
4. 重点失败节点分析与建议
5. 表格链接

其中：

- `overview_all.csv` 和 `top10_jobs_all.csv` 是 AI 播报的主输入
- `jobs_all.csv` 主要用于补充明细，不建议让 AI 在日报里过度展开

### 本地文件清理约定

当前发布脚本的真实行为是：

- 发布成功后自动清理本地中间产物
- 默认会删除：
  - `overview_all.csv`
  - `top10_jobs_all.csv`
  - `jobs_all.csv`

这样可以避免长期运行后本地磁盘持续膨胀。
