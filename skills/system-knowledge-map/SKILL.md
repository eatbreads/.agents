# System Knowledge Map

## Overview

把一份“已经写出来的技术报告”重新整理成“快速建立系统理解所需的知识地图”。
重点不是复述报告内容，而是帮助用户识别必须先补什么、应该按什么顺序学、每个领域该问哪些问题。

## Core Goal

默认围绕下面四件事输出：

1. 识别核心领域并按重要性排序
2. 为每个领域生成分层扫盲问题树
3. 总结最推荐的关键理解路径
4. 如果合适，再补一句最小知识闭环

## Working Rules

### 1. 先把报告看成“知识缺口地图”

不要先做摘要。
先识别：

- 这份报告真正依赖哪些技术领域
- 哪些概念被提到了，但没有解释
- 哪些链路是理解全局的主干
- 哪些系统边界决定了阅读难度

优先抽出“要理解报告，必须先知道什么”。

### 2. 核心领域必须按重要性排序

输出核心领域时，不要只列关键词。
每个领域都要说明“为什么重要”，例如：

- 它是不是整条主链路的基础
- 它是不是多个概念的前置知识
- 它是不是报告里频繁出现但默认读者已知的背景
- 它是不是系统边界、性能、稳定性或部署环境的关键

如果报告涉及很多领域，优先保留最影响理解主线的 3 到 6 个。

### 3. 问题树必须是“从扫盲到能落地理解”

每个领域都按下面四层组织：

- 基础认知
- 工作原理
- 工程实现
- 常见问题 / tradeoff

每个问题都要满足：

- 面试风格，可以口述回答
- 一问只讲一个知识点
- 问题范围不要过宽
- 能自然衔接到下一个问题

不要写成百科目录，也不要写成大而空的开放题。

### 4. 优先补报告里的“断层”

问题树优先覆盖这些内容：

- 报告里出现但没解释的术语
- 用户容易混淆的系统边界
- 主链路中的关键步骤
- 从现象跳到结论中间缺失的机制

典型边界包括：

- 本地环境 vs 远端环境
- 用户态 vs 内核态
- 构建阶段 vs 运行阶段
- 控制面 vs 数据面
- 调度层 vs 执行层

典型主链路包括：

- build
- mount
- 启动
- 调度
- 网络请求
- UT / 集成测试
- 数据读写

### 5. 学习路径必须强调依赖关系

“关键理解路径”不是把领域再列一遍。
要明确写出推荐顺序，并说明：

- 为什么先学它
- 它为后面的哪个领域提供基础
- 学到这里后，能看懂报告中的哪一段

目标是让用户知道“下一小时先补什么最值”。

### 6. 最小知识闭环要抓 20/80

如果报告主线清晰，就补一句“最小知识闭环”。
这一句要回答：

- 哪 20% 的知识最值得先掌握
- 为什么这部分能解释 80% 的报告

不要泛泛而谈，尽量点到具体链路或具体边界。

## Output Structure

默认按下面结构输出：

```markdown
## 第一步：核心领域
- 领域 1（为什么重要）
- 领域 2（为什么重要）

## 第二步：扫盲问题树
【领域：XXX】
一、基础认知
- 问题 1

二、工作原理
- 问题 2

三、工程实现
- 问题 3

四、问题与取舍
- 问题 4

## 第三步：关键理解路径
1. 先理解 XXX（因为它是基础）
2. 再理解 XXX（因为依赖前者）
3. 最后理解 XXX（形成闭环）

## 第四步：最小知识闭环
一句话总结
```

如果用户已经给了固定格式，优先服从用户格式。

## Quality Bar

输出前自查：

- 是否真的按重要性排序，而不是随手罗列
- 是否每个问题都足够小、足够具体
- 是否覆盖了报告里的关键边界和关键链路
- 是否能让用户按这个问题树继续学习，而不只是“看起来很完整”

## Style

- 用中文输出
- 语气像系统拆解教练，不像摘要机器人
- 优先帮助用户“建立理解框架”，不是“重复报告原文”
- 如果报告信息不足，要明确说明哪些判断是基于推断
