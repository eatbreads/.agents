# System Knowledge Map

把一份技术材料压缩成“按模块整理的纯问题清单”。

默认目标不是摘要，不是教学讲义，也不是分步骤分析，而是让用户能够快速扫一遍问题，知道这个系统最值得先搞懂什么。

## Default Output Goal

默认只做一件事：

- 输出按模块分组的高密度问题清单

除非用户明确要求，否则不要额外输出：

- 答案
- 解析
- 学习路径
- 最小知识闭环
- 第一步、第二步、第三步式流程
- 为什么这个模块重要的解释

用户给了固定格式时，优先服从用户格式。

## What To Extract First

先从材料里抽这几类信息，再出题：

- 核心对象：模块、组件、进程、线程、服务、仓库、目录、工具、运行单元
- 核心机制：调度、通信、存储、缓存、同步、编译、执行、容错、恢复
- 核心边界：本地 vs 远端、用户态 vs 内核态、控制面 vs 数据面、构建阶段 vs 运行阶段
- 核心链路：启动、调用、数据流转、失败处理、验证路径

抽这些信息的目的不是写分析，而是决定“应该分成哪些模块出题”。

## Preferred Output Shape

默认输出应该长得像这样：

```markdown
# 一、模块 A

* 问题 1
* 问题 2
* 问题 3

# 二、模块 B

* 问题 1
* 问题 2
* 问题 3
```

要求：

- 按模块分组
- 每组问题尽量覆盖该模块 60% 到 80% 的高频理解点
- 问题之间允许从基础到深入自然展开，但不要显式写“基础认知 / 工作原理 / 工程实现 / tradeoff”这些小标题，除非用户点名要
- 默认只给问题，不给答案

## Question Writing Rules

每个问题都要满足：

- 一问只问一个点
- 能口述回答
- 足够具体，不空泛
- 贴着当前材料，不要漂到泛化方法论
- 读完这一组问题后，用户能知道这个模块该从哪些角度理解

优先写这几类问题：

- 定义边界题
- 角色区别题
- 机制原理题
- 链路衔接题
- 失败归因题
- 设计取舍题

例如：

- `进程和线程的区别是什么？`
- `为什么线程切换通常比进程切换更快？`
- `共享内存为什么通常比消息队列更快？`
- `为什么本地编译通过，不代表线上一定能正常运行？`
- `某一步失败时，怎么判断是环境问题还是实现问题？`

## Default Module Strategy

优先按“用户最容易拿来扫盲”的方式分模块，而不是按作者写作顺序分模块。

常见分法包括：

- 基础对象
- 通信机制
- 同步与并发
- 调度
- 存储与内存
- 调用链路
- 异常与故障
- 工程验证

如果材料本身是一个具体系统，也可以改成：

- 组件角色
- 主调用链路
- 关键状态流转
- 运行环境边界
- 常见失败点
- 验证方式

目标是让模块标题本身就能帮助用户建立脑图。

## Strong Negative Rules

默认不要输出下面这些内容：

- `第一步 / 第二步 / 第三步`
- `关键理解路径`
- `最小知识闭环`
- `为什么这个领域重要`
- 大段摘要
- 大段背景铺垫
- 问题后面跟着答案
- 一问覆盖过大范围的百科题
- 脱离当前材料、放到任何系统都成立的空泛题

如果一个问题脱离当前材料依然完全成立，而且不能帮助用户缩小理解范围，那它大概率太泛了。

## Compression Rule

宁可少一点，也不要水。

默认做法：

- 保留最值得问的模块
- 每个模块列 4 到 10 个问题
- 总问题量以“用户 3 到 8 分钟能快速扫完”为目标

如果材料特别长，可以先压到 5 到 8 个模块，而不是把所有细枝末节都铺开。

## When Information Is Incomplete

如果材料信息不足，不要编造细节。

允许在最后补一个很短的 `补充说明`，只说明：

- 哪些模块是基于推断整理的
- 哪些关键链路在原文中信息不足

除这两点外，不要额外扩写。

## Style

- 用中文输出
- 风格像“高频面试问题整理 + 技术扫盲提纲”
- 默认无答案
- 默认高密度
- 默认按模块整理
- 优先让用户快速建立问题框架，而不是让内容显得完整
