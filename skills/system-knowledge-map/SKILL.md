# System Knowledge Map

## Overview

把一份“已经写出来的技术报告”重新整理成“帮助有工程基础的读者快速建立系统理解”的知识地图。
重点不是复述报告内容，也不是做零基础科普，而是帮助用户识别这套系统的对象分层、关键链路、环境边界、验证方式，以及接下来最该补的理解缺口。

## Core Goal

默认围绕下面四件事输出：

1. 识别核心领域并按重要性排序
2. 为每个领域生成分层问题树
3. 总结最推荐的关键理解路径
4. 如果合适，再补一句最小知识闭环

默认把用户视为“有通用工程经验，但对当前系统上下文不熟”的读者。
因此问题要优先服务于“快速建立系统模型”，而不是解释编程常识。

## Working Rules

### 1. 先把报告看成“知识缺口地图”

不要先做摘要。
先识别：

- 这份报告真正依赖哪些技术领域
- 哪些概念被提到了，但没有解释
- 哪些链路是理解全局的主干
- 哪些系统边界决定了阅读难度

优先抽出“要理解报告，必须先知道什么”。

### 2. 先抽对象、链路、边界，再出问题

生成问题前，先在脑子里完成这三步：

- 抽对象：仓库、目录、模块、组件、工具、运行单元
- 抽链路：开发、构建、集成、测试、部署、运行、验证
- 抽边界：本地 vs 远端、源码 vs 工作区、构建成功 vs 可运行、可跑 UT vs 可做端到端验证

如果没有先抽出这三类信息，就不要直接套四层模板。

问题树应该围绕“这些对象怎么串起来、边界在哪里、为什么这样分”来展开。

### 2. 核心领域必须按重要性排序

输出核心领域时，不要只列关键词。
每个领域都要说明“为什么重要”，例如：

- 它是不是整条主链路的基础
- 它是不是多个概念的前置知识
- 它是不是报告里频繁出现但默认读者已知的背景
- 它是不是系统边界、性能、稳定性或部署环境的关键

如果报告涉及很多领域，优先保留最影响理解主线的 3 到 6 个。

### 3. 问题树必须是“从扫盲到能落地理解”

每个领域都按下面四层组织：

- 基础认知
- 工作原理
- 工程实现
- 常见问题 / tradeoff

每个问题都要满足：

- 面试风格，可以口述回答
- 一问只讲一个知识点
- 问题范围不要过宽
- 能自然衔接到下一个问题
- 问完以后，能帮助用户继续推演系统主线

不要写成百科目录，也不要写成大而空的开放题。

这里的“基础认知”不是给零基础用户补常识，而是帮助读者先分清：

- 谁是主仓库，谁是下游集成仓库
- 哪个目录是源码，哪个目录是工作区依赖
- 哪一步属于构建，哪一步属于运行
- 哪些失败是环境边界导致，哪些才可能是实现问题

### 4. 优先问“系统拆解题”，不要问“通用扫盲题”

优先生成下面这几类问题：

- 对象角色题
  例：`tos-fuse、bytefuse、storage/ 分别是什么角色？`
- 分层关系题
  例：`为什么 storage/ 看起来像仓库，但其实不是主开发仓？`
- 链路衔接题
  例：`tos-fuse 改完以后，为什么还要经过 bytefuse？`
- 环境边界题
  例：`为什么 devbox 能编代码，不代表它能跑 FSX 客户端？`
- 失败归因题
  例：`UT 失败时，怎么判断是代码问题还是环境问题？`
- 验证路径题
  例：`如果要验证自己改过的代码，为什么要走 scm_tos_fuse -> bytefuse -> ECS 安装验证 这条链路？`

### 5. 明确避免这几类差问题

默认不要生成下面这些问题，除非用户明确要求从零讲起：

- 纯常识题
  例如：`为什么要配置 SSH？`
- 过于空泛的方法论题
  例如：`为什么这份报告里要同时关注构建和测试？`
- 报告原文已经直接回答的问题
- 与主链路关系很弱的流程碎问题
- 一问覆盖太大范围、无法口述回答的问题

如果一个问题脱离当前系统，放到任何项目里都成立，那它大概率太泛了。

### 6. 优先补报告里的“断层”

问题树优先覆盖这些内容：

- 报告里出现但没解释的术语
- 用户容易混淆的系统边界
- 主链路中的关键步骤
- 从现象跳到结论中间缺失的机制

典型边界包括：

- 本地环境 vs 远端环境
- 用户态 vs 内核态
- 构建阶段 vs 运行阶段
- 控制面 vs 数据面
- 调度层 vs 执行层

典型主链路包括：

- build
- mount
- 启动
- 调度
- 网络请求
- UT / 集成测试
- 数据读写

除了术语，还要优先覆盖这些“系统理解断层”：

- 从“在哪改代码”到“怎么进下游集成”的断层
- 从“本地编通”到“为什么还不能正式运行”的断层
- 从“UT 失败”到“如何判断失败归因”的断层
- 从“目录很多”到“谁是源码、谁是工作区、谁是产物”的断层

### 7. 学习路径必须强调依赖关系

“关键理解路径”不是把领域再列一遍。
要明确写出推荐顺序，并说明：

- 为什么先学它
- 它为后面的哪个领域提供基础
- 学到这里后，能看懂报告中的哪一段

目标是让用户知道“下一小时先补什么最值”。

### 8. 最小知识闭环要抓 20/80

如果报告主线清晰，就补一句“最小知识闭环”。
这一句要回答：

- 哪 20% 的知识最值得先掌握
- 为什么这部分能解释 80% 的报告

不要泛泛而谈，尽量点到具体链路或具体边界。

## Question Selection Heuristics

如果用户像这次一样已经有工程经验，优先多问下面这些：

- 角色和边界
- 链路为什么这样串
- 哪一步依赖前一步
- 为什么某个环境只能做到 A 做不到 B
- 某类失败信号背后对应哪类限制

少问下面这些：

- 编程或 Linux 常识
- 能从报告表面直接读出来的句子改写题
- 不影响后续开发决策的问题

## Report-to-Questions Mapping

拿到原始材料后，优先从材料里提炼这四类信息，再据此出题：

1. 关键对象
   例如仓库、目录、工具、运行介质、测试类别
2. 关键链路
   例如拉代码 -> 编译 -> 跑 UT -> 下游集成 -> ECS 验证
3. 关键边界
   例如 devbox vs ECS、源码仓 vs Blade 工作区、UT 可跑 vs 无法验证挂载
4. 关键失败信号
   例如 `HeadBucket failed`、`Couldn't resolve proxy name`、`Read-only file system`

问题树应该尽量贴着这四类信息生成，而不是脱离原文自由发挥。

## Output Structure

默认按下面结构输出：

```markdown
## 第一步：核心领域
- 领域 1（为什么重要）
- 领域 2（为什么重要）

## 第二步：扫盲问题树
【领域：XXX】
一、基础认知
- 问题 1

二、工作原理
- 问题 2

三、工程实现
- 问题 3

四、问题与取舍
- 问题 4

## 第三步：关键理解路径
1. 先理解 XXX（因为它是基础）
2. 再理解 XXX（因为依赖前者）
3. 最后理解 XXX（形成闭环）

## 第四步：最小知识闭环
一句话总结
```

如果用户已经给了固定格式，优先服从用户格式。

## Quality Bar

输出前自查：

- 是否真的按重要性排序，而不是随手罗列
- 是否每个问题都足够小、足够具体、足够贴着当前系统
- 是否覆盖了报告里的关键边界和关键链路
- 是否避免了“任何系统都能问”的泛问题
- 是否能让用户按这个问题树继续学习，而不只是“看起来很完整”

## Style

- 用中文输出
- 语气像系统拆解教练，不像摘要机器人
- 优先帮助用户“建立理解框架”，不是“重复报告原文”
- 默认面向“懂工程、但不懂这个系统”的读者
- 如果报告信息不足，要明确说明哪些判断是基于推断
